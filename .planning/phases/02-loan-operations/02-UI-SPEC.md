## Spacing Scale

Declared values (must be multiples of 4). Source: established in Phase 1 codebase — `p-6 space-y-4 gap-4 py-16` pattern observed across pages.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Compact element spacing, form field gaps |
| md | 16px | Default element spacing, card padding |
| lg | 24px | Page section padding (`p-6`) |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Empty state vertical padding (`py-16` = 64px exception — see below) |
| 3xl | 64px | Page-level vertical spacing |

Exceptions:
- Empty state vertical padding: `py-16` (64px top + bottom) — matches Phase 1 loans page pattern exactly
- Touch targets for action buttons in payment table rows: minimum 44px height — use `h-11` on inline action triggers
- Print receipt layout: spacing tokens are irrelevant on `@media print` — receipt uses fixed `pt-8 pb-4` print margins

## Typography

Source: Established in Phase 1 codebase. All pages use `text-2xl font-semibold` for page headings, `text-sm text-muted-foreground` for secondary text. Geist Sans is the system font.

| Role | Tailwind Class | Size | Weight | Line Height | Usage |
|------|---------------|------|--------|-------------|-------|
| Display | `text-2xl font-semibold` | 24px | 600 | 1.2 | Page headings (Payments, Receipt) |
| Heading | `text-lg font-semibold` | 18px | 600 | 1.2 | Card headings, section titles, receipt business name |
| Body | `text-sm` (default weight) | 14px | 400 | 1.5 | Table cells, form labels, body copy |
| Label | `text-xs text-muted-foreground` | 12px | 400 | 1.5 | Secondary labels, timestamps, receipt field names |

Mono variant: `font-mono text-xs` — used for loan reference IDs, receipt numbers (e.g. `LOAN-A3B2C1D4`, `PAY-F1E2D3C4`)

Maximum 4 type sizes in use. Do not introduce additional sizes.

## Color

Source: `src/app/globals.css` CSS custom properties (oklch). Phase 1 established token usage patterns.

| Role | CSS Token | Light Value | Usage |
|------|-----------|-------------|-------|
| Dominant (60%) | `--background` | `oklch(1 0 0)` = white | Page background, receipt background |
| Secondary (30%) | `--card` / `--sidebar` | `oklch(1 0 0)` / `oklch(0.985 0 0)` | Payment list cards, sidebar, nav, form surfaces |
| Accent (10%) | `--primary` | `oklch(0.205 0 0)` = near-black | See reserved-for list below |
| Muted | `--muted-foreground` | `oklch(0.556 0 0)` = medium grey | Secondary text, empty state copy, disabled states |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` = red | Delete payment button, soft-deleted row strikethrough indicator |
| Border | `--border` | `oklch(0.922 0 0)` = light grey | Table dividers, card borders, input borders |

**Accent (`--primary`) reserved for:**
- Primary action buttons: "Record Payment", "Print Receipt", "Save Changes"
- Active navigation item in sidebar
- Form submit buttons

**Accent is NOT used for:**
- Informational badges (use `--secondary` / badge `variant="secondary"`)
- Table row highlights
- Tooltips or hover states on non-primary actions

**Soft-deleted payment rows:** Apply `opacity-60` and `line-through` text decoration to all cells in a deleted payment row. Do not use a separate background color — the strikethrough communicates deletion clearly. Source: CONTEXT.md — "Deleted payments appear crossed out in the payment list."

**Receipt page:** White background only, no dark mode rendering. Add `print:bg-white` and ensure receipt container never picks up dark mode variables. Source: CONTEXT.md — print-optimized layout.

### Disbursement Receipt Page — `/receipts/disbursement/[loanId]` (RCPT-01, RCPT-03)

| Component | Source | Notes |
|-----------|--------|-------|
| Button | installed | "Print Receipt" — calls `window.print()` |
| alert | `npx shadcn add alert` | RCPT-03 blocked state when required fields missing |

**Print button placement:** Outside receipt body, inside a `print:hidden` wrapper div. Aligned right, above the receipt.

**Blocked state:** When RCPT-03 triggered, print button is `disabled`. An Alert component (variant `destructive`) appears above the button listing which fields are missing. Copy: see Copywriting Contract.

**Receipt fields to render:** Business name/logo, receipt number (`LOAN-{id.slice(0,8).toUpperCase()}`), date, customer name, customer contact, loan amount (UGX), interest rate (% per month), minimum interest period (days), collateral details, issued-by officer name.

### Repayment Receipt Page — `/receipts/repayment/[paymentId]` (RCPT-02, RCPT-03)

Same print/blocked structure as disbursement receipt.

**Receipt fields to render:** Business name/logo, receipt number (`PAY-{id.slice(0,8).toUpperCase()}`), date, customer name, loan reference (`LOAN-{loanId.slice(0,8).toUpperCase()}`), payment amount (UGX), interest paid (UGX), principal paid (UGX), outstanding balance after payment (UGX), received-by officer name.

### Print Layout CSS Contract

```css
/* Applied to receipt pages only */
@media print {
  .print-hidden { display: none !important; }
  body { background: white; color: black; }
  .receipt-body {
    max-width: 100%;
    font-family: var(--font-geist-sans), sans-serif;
    padding: 2rem;
  }
  /* Remove card shadows and borders for clean print */
  .receipt-body * { box-shadow: none !important; border-color: #ccc !important; }
}
```

Use Tailwind `print:hidden` utility on all non-receipt UI elements (nav, sidebar, header, back links, print button).

## Interaction States

Every interactive element must implement all four states. No exceptions.

| State | Visual Treatment |
|-------|-----------------|
| Default | Token-based colors as specified above |
| Hover | shadcn built-in hover tokens — do not override |
| Loading / Pending | Button: `disabled` prop + spinner icon (Loader2 from lucide) replacing button text |
| Disabled | 50% opacity via `disabled:opacity-50` (shadcn default) |
| Error | Input: red ring via `aria-invalid`, error message in `text-destructive text-xs` below field |
| Success | sonner toast: "Payment recorded successfully" — auto-dismiss 4s |

**Form submission flow:**
1. User clicks "Record Payment" — button enters loading state (spinner + "Recording…" text)
2. Server Action resolves — success: toast + form resets + page revalidates
3. Server Action rejects — error toast + form stays populated for correction (no data loss on error)

**Payment delete flow:**
1. User opens action menu → clicks "Delete Payment"
2. Dialog opens with reason textarea and confirmation button
3. User types reason → confirm button enables
4. User clicks confirm → button enters loading state
5. Success: dialog closes + toast "Payment deleted" + row updates to soft-deleted visual

## Email Notification Templates (ALRT-02)

Delivered via Resend. Plain HTML — no design framework. Fire-and-forget from Server Actions.

### Shared template structure

```
Subject: [EventType] — Loan {LOAN-XXXXXXXX} — {Amount} UGX

Body:
  Event:     Payment Recorded / Payment Updated / Payment Deleted / Loan Disbursed
  Actor:     {user name} ({user email})
  Loan ref:  LOAN-{id.slice(0,8).toUpperCase()}
  Amount:    UGX {formatted amount}
  Timestamp: {ISO 8601 datetime, Africa/Kampala timezone}

  This is an automated notification from the lending system.
```

**Subject line patterns:**
- Payment created: `Payment recorded — Loan LOAN-XXXXXXXX — UGX 500,000`
- Payment updated: `Payment updated — Loan LOAN-XXXXXXXX — UGX 500,000`
- Payment deleted: `Payment deleted — Loan LOAN-XXXXXXXX — UGX 500,000`
- Loan disbursed: `Loan disbursed — LOAN-XXXXXXXX — UGX 2,000,000`

**Recipient:** All active users with role `admin` or `superAdmin` — queried dynamically from DB. Source: RESEARCH.md open question #1 resolved to dynamic DB query.
