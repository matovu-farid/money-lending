   - pgEnum `loanStatusEnum` with values: "pending", "active", "partially_paid", "fully_paid", "defaulted"
   - Table name: `loans`
   - Columns:
     - `id`: uuid PK, defaultRandom
     - `customer_id`: uuid, notNull (references customers.id)
     - `principal_amount`: numeric({ precision: 15, scale: 2 }), notNull — DO NOT use `mode: 'number'`

     - `term_days`: integer, notNull, default(30)
     - `min_interest_days`: integer, notNull, default(30) — overridable per loan (LOAN-11)
     - `start_date`: timestamp with timezone, notNull
     - `due_date`: timestamp with timezone, notNull — computed as start_date + term_days at issuance
     - `status`: loanStatusEnum, notNull, default("pending")
     - `interest_rate_override`: numeric({ precision: 5, scale: 4 }), nullable — admin override (LOAN-11)
     - `min_period_override`: integer, nullable — admin override (LOAN-11)
     - `issued_by`: text, notNull — user ID of the officer who issued
     - `created_at`: timestamp with timezone, defaultNow, notNull
     - `updated_at`: timestamp with timezone, defaultNow, notNull
   - Table name: `collateral`
   - Columns: `id` (uuid PK), `loan_id` (uuid, notNull, references loans.id), `nature` (text, notNull — e.g. "land_title", "vehicle_log_book"), `description` (text, nullable), `created_at` (timestamp with timezone, defaultNow, notNull)

1. Create `src/lib/errors.ts` — all Effect.js tagged error types used across the service layer:
   ```typescript
   import { Data } from "effect"

   // Database errors
   export class DatabaseError extends Data.TaggedError("DatabaseError")<{ cause: unknown }> {}

   // Entity not found errors
   export class CustomerNotFound extends Data.TaggedError("CustomerNotFound")<{ id: string }> {}
   export class LoanNotFound extends Data.TaggedError("LoanNotFound")<{ id: string }> {}

   // Validation errors
   export class ValidationError extends Data.TaggedError("ValidationError")<{ message: string; field?: string }> {}
   export class IncompleteLoanRequirements extends Data.TaggedError("IncompleteLoanRequirements")<{
     missing: string[]
   }> {}

   // Auth errors
   export class UnauthorizedError extends Data.TaggedError("UnauthorizedError")<{ reason: string }> {}
   export class ForbiddenError extends Data.TaggedError("ForbiddenError")<{ action: string; role: string }> {}

   // Business rule errors
   export class DuplicateError extends Data.TaggedError("DuplicateError")<{ entity: string; field: string }> {}
   ```

2. Create `src/types/index.ts` — shared TypeScript types for API request/response shapes:
   ```typescript
   import type { InferSelectModel, InferInsertModel } from "drizzle-orm"
   import type { customers } from "@/lib/db/schema/customers"
   import type { loans, collateral } from "@/lib/db/schema/loans"
   import type { auditLog } from "@/lib/db/schema/audit"

   // Inferred DB row types
   export type Customer = InferSelectModel<typeof customers>
   export type NewCustomer = InferInsertModel<typeof customers>
   export type Loan = InferSelectModel<typeof loans>
   export type NewLoan = InferInsertModel<typeof loans>
   export type Collateral = InferSelectModel<typeof collateral>
   export type NewCollateral = InferInsertModel<typeof collateral>
   export type AuditLogEntry = InferSelectModel<typeof auditLog>

   // Loan status type
   export type LoanStatus = "pending" | "active" | "partially_paid" | "fully_paid" | "defaulted"

   // Customer status type
   export type CustomerStatus = "active" | "blacklisted" | "inactive"

   // Role hierarchy
   export const ROLE_LEVELS = {
     unassigned: 0,
     loanOfficer: 1,
     admin: 2,
     superAdmin: 3,
   } as const
   export type UserRole = keyof typeof ROLE_LEVELS

   // API response envelope
   export type ApiResponse<T> = { data: T } | { error: string; details?: unknown }
   ```

   ```markdown
   ### INFR-06 Layer Injection — Deliberate Deferral

   INFR-06 requires `Effect<Success, Error, Dependencies>` with dependencies injected via Layer. For Phase 1, service functions use `Effect<S, E, never>` with `db` closed over from module scope (no Layer injection). This is a deliberate architectural choice per RESEARCH.md Open Question #3: introducing Layers in Phase 1 adds complexity without benefit given the small service graph. Layer injection will be introduced in Phase 2 when the service dependency graph grows. Phase 1 satisfies the "Effect.js throughout service layer" and "typed error channels" aspects of INFR-06 but defers the "dependencies injected via Layer" aspect.
   ```
