Open-Source Real-Time Chat Libraries

No third-party services — self-hosted, works with your existing PostgreSQL + Next.js stack

A

Socket.io

The classic. Full-duplex WebSocket with automatic fallback to long-polling.

  • Most mature — huge ecosystem, tons of examples
  • Built-in rooms, namespaces, broadcasting
  • ~10 KB client bundle
  • Caveat: Requires a custom Node.js server — breaks Vercel serverless deployment
  • socket.io + socket.io-client
B

PartyKit / PartyServer

Edge-native WebSockets. Open-source, Cloudflare Workers-based.

  • ~5 KB client bundle (thinnest option)
  • usePartySocket React hook included
  • Free on Cloudflare's free tier
  • Caveat: Separate Cloudflare deployment — two infra targets to manage
  • partykit + partysocket
C

ws + Next.js Custom Server

Raw WebSocket. Minimal abstraction — just the Node.js ws library attached to Next.js.

  • Zero overhead — no framework, just WebSocket protocol
  • ~2 KB (uses native browser WebSocket API, no client lib needed)
  • Full control over protocol and message format
  • Caveat: You build rooms, reconnection, heartbeat yourself. Also needs custom server.
  • ws
D

Socket.io + Standalone Server (recommended)

Best of both worlds. Socket.io running as a lightweight standalone service alongside Next.js — no custom Next.js server needed.

  • Next.js stays serverless / standard — no custom server.js
  • Socket.io runs in a small separate Node process (e.g. chat-server.ts)
  • Shares the same PostgreSQL DB via Drizzle
  • Built-in rooms, presence, reconnection, broadcasting
  • Deploy both with a single docker-compose or process manager
  • socket.io + socket.io-client

💡 My Recommendation: D

Socket.io as a standalone service keeps your Next.js app untouched while giving you the most battle-tested WebSocket library. It shares your Drizzle/PostgreSQL stack, and the two processes deploy together easily. You get rooms, reconnection, and broadcasting for free — no need to build those from scratch like with raw ws.
